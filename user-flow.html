review.nvim User Flow

A reviewer journey map for moving from opening a review to publishing or exporting actionable feedback.

Entry points Navigation decisions Review actions Context capture Output workflow

Primary Review Journey

Open Review :Review creates the review tab and loads local, ref, PR/MR, or GitButler context. Next: scan scope
Choose Work Unit Use Files for all changes, or scope rows for commit/GitButler units. Next: inspect diff
Read Diff Move through hunks/files, toggle split, return between rail and diff with s. Next: decide
Need Feedback? No issue: mark reviewed. Issue found: add note or suggestion with selected code. Next: manage queue
Publish / Export Stage notes, publish to forge, or copy/export a handoff packet.

1. Start And Scope

Open session:Review

Resolve review target, detect GitButler workspace, load files, commits, remote threads, and local notes.

Pick mode<Tab>

All changes, current commit, or select commit. GitButler maps this to branch/unassigned review units.

Compare units?C

When units overlap or depend on each other, open a comparison before reviewing line-by-line.

Open detailsK

Inspect commit or review-unit metadata before deciding where to start.

2. Navigate Review Surface

Open file<CR>

From Files, select a file and move into the diff pane.

Move through code]c

Jump hunks and files while the cursor stays on code, not the diff gutter.

Switch layouts

In the diff, toggle unified versus split. In Files/Threads, return to the last diff/split position.

Check thread queuet

Move to Threads to inspect local notes, remote discussions, stale context, and review timeline.

3. Create Feedback

Select exact codeV

Use native visual-line selection or visual block. Gutter numbers and +/- signs are excluded from copied code.

Add notea

Create a line or range note from the current cursor or visual selection.

Suggest patchS

Create a suggestion note with a GitHub-compatible suggestion block.

Add evidenceB

Open blame, attach selected blame context, or open file history with L.

4. Manage And Deliver

Open queueN

Review all draft, staged, resolved, published, local, and remote note items.

Stage feedbacks

Toggle notes between draft and staged so only ready feedback is published.

Publish or handoffP

Publish staged notes to GitHub/GitLab, or copy a full/unit/local handoff packet.

Clean upd

Delete local notes, resolve/reopen notes, clear local feedback, or refresh stale remote context.

Common UX Paths

Quick Review :Review -> file -> hunks -> mark reviewed.
Comment Flow Diff -> select lines -> a -> draft note -> notes list.
Suggestion Flow Diff -> visual range -> S -> edit suggestion -> stage.
Context Flow B blame or L history -> attach row -> add note.
Delivery Flow N notes -> stage -> publish, export, or copy handoff.

Decision Outcomes

No issue found

Mark file/unit reviewed, hide reviewed files if needed, continue to next file.

Issue needs explanation

Add a note, optionally attach blame/history context, leave as draft until reviewed.

Issue has exact fix

Create a suggestion note from selected code and stage it for publishing.

Remote context changed

Refresh review data, stale notes/threads move into stale sections instead of disappearing.

Change Review Base Flow

Start In Local Review Base changes are available for local git review sessions. GitButler uses its workspace merge base. Trigger
Open Base Prompt Press <leader>B or run :ReviewChangeBase. Choose ref
Enter Base Ref Type a branch, tag, commit, or relative ref such as main, HEAD~2, or origin/main. Rebuild
Refresh Diff review.nvim re-diffs the active session against the new base and preserves relevant review state. Continue
Review Against New Base Files, threads, and notes now reflect the updated comparison target.

Compare Commits / Review Units Flow

Expose Units Use <Tab> until commit or GitButler review-unit rows are visible in the rail. Pick left
Select Current Unit Press <CR> on the commit/unit you want as the left side of the comparison. Compare
Choose Compare Press C. On a different unit row, that row becomes the right side. Otherwise a picker opens. Inspect
Read Unit Diff The comparison popup shows overlap, left-only files, and right-only files for the two units. Act
Return To Review Use the result to choose review order, identify duplicated changes, or add targeted feedback.
The flow emphasizes UX intent: rails are for orientation and queue navigation, diff/split is for code decisions, context panels collect evidence, and the notes list is the delivery checkpoint.
